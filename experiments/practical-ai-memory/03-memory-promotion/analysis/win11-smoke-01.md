# Win11 Smoke 01-02：MCP 片段读取漏记

前两批 Win11 Smoke 都因 MCP 工作区读取漏记而停止并隔离，没有进入 45 次正式矩阵。

## 现象

- PowerShell 工作区读取被 Windows 沙箱拒绝。
- 模型随后通过 `node_repl` MCP 调用恢复读取并正确回答。
- 两次 MCP 结果明确包含夹具内容片段，但指标分类器没有把它们计为工作区读取。
- 元数据因此错误地显示工作区 MCP 调用为 0，同时把指标覆盖标记为完整。

答案正确不能弥补过程指标漏记。该运行已经隔离，不进入有效 Smoke 或正式结论。

## 根因

分类器原本支持整份夹具内容和 JSON 包装的多文件结果，但要求完整文件原文出现在 MCP 结果中。第一次修正允许 MCP 结果本身是夹具内容片段；第二批 Smoke 进一步证明实际结果还可能在片段前后附加工具包装文本，因此仍被静默归类为非工作区调用。

只读结构调查确认：两份代表运行共有 4 次 `node_repl` 调用，均通过 `result.content[].text` 返回模型实际读取的文本，且都包含可确认的夹具连续片段。问题不在结果提取或字节口径，而在夹具片段匹配方式。

## 修正

分类器保留整文件精确匹配，并增加最少 32 个 UTF-8 字节的夹具连续片段搜索。短标签或普通词重合不计为文件读取，避免把无关 MCP 输出误判为工作区内容。

如果 `node_repl` 参数明确包含文件系统读取操作，但结果仍无法和夹具内容或路径对应，分类器现在保守标记为 `unknown`，使指标覆盖门禁失败，不再静默记为非工作区调用。

新增回归测试覆盖：

- 结构化 MCP 结果中的长夹具片段应计入工作区调用和输出字节。
- 前后包含工具包装文本的夹具片段仍应被识别。
- 短文本与夹具重合时不得误判。
- 无法归属的 `node_repl` 文件读取必须标记为覆盖不完整。

修正后必须从头执行完整的 15 次 Win11 Smoke。两批隔离运行都不能复用；只有新 Smoke 的环境隔离、语义答案和指标覆盖全部通过后，才能启动正式矩阵。
