# macOS 与 Win11 正式矩阵对照

## 对照范围

两个平台都使用相同冻结夹具、提示、评分表和模型配置：

- 每个平台 36 次正式运行，共 72 次
- 模型：`gpt-5.6-sol`
- 推理强度：`medium`
- Codex CLI：`0.144.1`
- 插件：显式关闭
- 每次运行：`runtime_tool_access_calls=0`
- 每个任务与条件：`n=3`，`workspace_metrics_n=3`

数据入口：

- [`formal-macos-gpt-5.6-sol-medium.json`](../data/formal-macos-gpt-5.6-sol-medium.json)
- [`formal-win11-gpt-5.6-sol-medium.json`](../data/formal-win11-gpt-5.6-sol-medium.json)

Win11 只回传脱敏聚合文件，私有原始事件、评分文件、临时路径和会话标识仍留在 Windows 设备。

有效扣分：

- macOS 仅索引：三次高后果任务均缺少“将公开内容修正为脱敏证据”的独立替代方案，各扣 1 分。
- Win11 选择性常驻与仅索引：各有一次相同高后果修正项遗漏。
- Win11 仅索引：一次低频参考资料任务在读取被沙箱拒绝后没有恢复，最终未回答，得到 `0/4`。

所有扣分样本均保留为 `protocol_valid=true`。两个平台都没有发现无依据断言或无关项目事实。

## Windows 工具链观察

Win11 侧只读审计报告记录：

- 36 次均出现当前 CLI 不支持 PowerShell shell snapshot 的统一警告。
- 24 次需要读取文件的运行出现工作区相对读取被 Windows 沙箱拒绝。
- 23 次通过后续重试恢复并正确作答，1 次没有恢复。
- 20 份私有答案包含隔离临时文件链接；这些内容没有回传，也不进入公开证据。

聚合数据能够看到对应后果：Win11 按需组的工作区调用和输出字节明显增加。由于 macOS 没有出现同类拒绝，不能把差异只归因于记忆结构。

## 结论边界

两个平台共同支持：

1. 全部常驻能消除当前静态夹具中的项目文件读取，但持续支付固定上下文成本。
2. 选择性常驻不能保证减少检索，也没有在 Win11 完全消除高后果遗漏。
3. 仅索引的效果依赖检索链路；在 macOS 减少上下文，在 Win11 则受到沙箱拒绝和重试影响。
4. 信息加载策略不能只按 token 或字节选择，还要考虑变化频率、遗漏后果、检索可靠性和工具链差异。

本实验不能证明全部常驻适合真实大型项目，因为它没有直接测量入口膨胀、陈旧状态、副本冲突和长期维护成本。
