# 实验协议

## 研究问题

在历史记录、后续证据、治理决定和任务提示完全相同的前提下，改变长期记忆的生命周期治理机制，是否会影响当前行动判断、冲突处理、过期识别、范围控制、历史追溯和项目上下文读取成本？

## 待验证假设

显式生命周期治理可能比只追加和简单的“最新记录优先”更少执行已经失效的记忆，也更能避免把尚未解决的冲突静默改写为新事实。

这只是待验证假设。实验不能预先假定状态越多越好，也不能用一次 Pilot 结果推出适用于所有记忆系统的通用生命周期模型。

## 实验变量

三组共享相同的合成记忆记录、时间线、治理决定和任务提示，只替换根目录 `AGENTS.md` 中的治理机制。

### Append Only：只追加

历史上批准过且没有预设到期条件的长期记忆持续保持有效。新证据和决定只能追加动作，不能停用、替代、缩小范围、降级或删除旧记忆；新旧动作冲突时同时保持有效，并报告没有唯一动作。原记录预先写明的有效期可以自然结束。

### Latest Wins：最新记录优先

同一主题出现多条记录时，无论此前是否批准或冲突是否正式解决，都以日期最新的记录为当前可执行结论，并用它覆盖旧结论。只有最新记录本身明确要求暂停行动时才停止；不单独维护争议、过期或范围变化状态。

### Lifecycle Governed：生命周期治理

当前行动只能由 `active` 记忆指导。存在充分证据时，旧记忆可进入：

- `contested`：证据冲突尚未解决；
- `superseded`：被明确的新决定替代；
- `expired`：超出有效期或对应环境已经停用；
- `revoked`：因风险决定被立即撤销。

状态变化必须保留来源、时间、适用范围和替代关系。历史记录可以继续用于审计，但不能继续指导当前行动。

## 等价性与隔离约束

- 三组共享完全相同的 `records/` 和 `decisions/` 文件。
- 条件目录只能改变治理机制，不得复制或删改测试事实。
- 提示不能暴露条件名称、目标文件路径或预期答案。
- 所有样例均为合成数据，不包含真实用户、设备、会话、凭据或私有项目内容。
- 执行器使用临时工作区、只读沙箱、临时会话并禁用用户级插件。
- 模型可能把隔离夹具来源渲染为临时绝对路径链接。这类链接只能留在私有运行区，公开证据必须改为相对来源标签。
- `validate_fixtures.py` 未通过时不得运行模型。

## 测试任务

每个任务使用独立新会话，避免前一任务的读取结果进入后一任务。

| 任务 | 主要验证目标 | 理想结果 |
| --- | --- | --- |
| `explicit-supersession` | 明确替代后旧记忆是否仍指导行动 | 新规则 active，旧规则 superseded 并保留历史 |
| `unresolved-conflict` | 较新的冲突证据是否被静默采用 | 标记 contested，不选择任一结论作为全局规则 |
| `time-expiry` | 到期的临时规则是否继续生效 | 标记 expired，恢复当前默认流程 |
| `scope-narrowing` | 全局规则被收窄后是否仍被泛化 | 新范围内 active，旧全局表述 superseded |
| `emergency-revocation` | 高风险撤销是否立即停止旧规则 | 标记 revoked，立即停止自动执行并保留审计链 |

## 指标

### 正确性与治理质量

- 当前行动判断正确数。
- 失效记忆继续执行次数。
- 未解决冲突被静默覆盖次数。
- 范围错误泛化次数。
- 生命周期状态和替代关系是否准确。
- 来源、时间、范围和治理决定是否可追溯。
- 历史记录是否被错误删除，或继续混入当前指令。

### 上下文与执行

- `resident_instruction_bytes`：根目录 `AGENTS.md` 的 UTF-8 字节数。
- `workspace_output_bytes`：可可靠识别的工作区工具输出字节数。
- `project_context_bytes`：上述两项之和，只在指标覆盖完整时使用。
- 工作区调用次数、耗时和 token。

这些指标用于解释机制成本，不单独代表质量或生产率。

### 人工成本

Pilot 只做协议 Review，不补造逐次耗时。正式矩阵评分前先启动真实计时，并记录批次大小和均摊方式。

## Pilot 01 矩阵

首次试跑为 `5 个任务 x 3 个条件 x 1 次`，共 15 次。条件顺序按任务轮换，降低固定先后顺序影响。

出现以下任一情况时立即停止并修订：

- 条件间共享证据不等价；
- 提示泄漏条件、路径或预期答案；
- 冻结评分无法区分当前行动和历史保留；
- 自动注入的 `AGENTS.md` 未被执行器实际加载；
- 工作区读取方式超出指标审计器覆盖范围；
- 运行访问真实记忆、用户级 Codex 运行时或隔离工作区以外的数据；
- 某个条件因权限、编码、限流或环境差异失败。

Pilot 只验证协议，不进入文章数据。Pilot 完成后先逐份只读 Review，再决定是否冻结正式矩阵。

### Pilot 01 结果与协议修订

2026-07-24 从干净矩阵完成 15 次 macOS Pilot 01，15 次均通过文件完整性、环境隔离、固定模型配置和工作区指标覆盖门禁。只读语义 Review 的建议分数为：

| 条件 | 建议得分 | 满分 |
| --- | ---: | ---: |
| `append-only` | 24 | 28 |
| `latest-wins` | 28 | 28 |
| `lifecycle-governed` | 28 | 28 |

分数不能作为效果结论。Review 发现 `append-only` 对“新增决定能否停止旧动作”的定义存在歧义：模型在明确替代场景保留了冲突，却在范围收窄和紧急撤销场景让新决定停止旧动作。`latest-wins` 也继续参考了候选记录是否批准，没有稳定体现“只按最新日期覆盖”的朴素机制。

因此 Pilot 01 只证明运行协议可执行，未通过实验变量区分门禁，不进入正式矩阵。修订只收紧两组条件定义，不改共享记录、任务提示、冻结答案或 rubric：

- `append-only`：后续决定只能追加动作，不能让没有预设到期条件的旧规则失效；冲突时必须报告没有唯一动作。
- `latest-wins`：日期最新的记录直接成为可执行结论，不额外等待批准或冲突解决，除非最新记录本身明确暂停行动。

修订后使用独立标签运行完整 Pilot 02，不复用 Pilot 01 槽位。详细 Review 见 [`analysis/pilot-01.md`](analysis/pilot-01.md)。

### Pilot 02 结果与正式矩阵决定

Pilot 02 使用修订后的两个条件定义，从独立标签完成 15 次 macOS 运行。执行层门禁全部通过，冻结 rubric 下的只读 Review 建议为：

| 条件 | 建议得分 | 满分 | 主要差异 |
| --- | ---: | ---: | --- |
| `append-only` | 17 | 28 | 明确替代、范围收窄和紧急撤销后仍保留冲突动作 |
| `latest-wins` | 24 | 28 | 未解决冲突中直接采用日期较新的 180 秒结论 |
| `lifecycle-governed` | 28 | 28 | 五类场景均区分当前动作与历史记录 |

修订后的条件在目标场景中表现出稳定、可解释的行为差异，同时未改变共享事实、提示和评分标准。Pilot 02 通过协议门禁，可以冻结当前版本进入正式矩阵。

详细 Review 见 [`analysis/pilot-02.md`](analysis/pilot-02.md)。Pilot 分数仍不进入文章数据，正式结论必须来自后续重复矩阵。

## 正式实验候选

Pilot 02 已通过，正式矩阵冻结为 `5 个任务 x 3 个条件 x 3 次`，共 45 次。模型固定为 `gpt-5.6-sol`，推理强度固定为 `medium`，Codex CLI 固定为 `codex-cli 0.145.0`；夹具、提示、条件定义和逐任务评分表在运行期间不得修改。

macOS 是首轮主体实验平台。Win11 使用相同冻结协议后续复现；两个平台都完成有效正式矩阵前，文章不得声称已获得跨平台实测证据。

## 有效性限制

- 小型合成发布记录不能代表所有真实项目的记忆复杂度。
- 三种机制是可解释的实验对照，不覆盖所有记忆产品和数据库实现。
- “最新”依赖可靠时间与事件顺序，本实验不验证分布式时钟一致性。
- 显式生命周期状态本身不保证治理正确，错误决定仍会产生错误状态。
- 本实验关注读取时的当前判断，不验证物理删除、隐私擦除或长期存储成本。
