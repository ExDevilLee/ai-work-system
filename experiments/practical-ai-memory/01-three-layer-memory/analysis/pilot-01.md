# Pilot 01 分析

日期：2026-07-17

结论：运行与证据采集链路通过，但当前协议不适合直接进入正式重复实验。

## 结果

| 指标 | Baseline | Layered |
| --- | ---: | ---: |
| 正确性 | 10/10 | 10/10 |
| 无依据断言 | 0 | 0 |
| 工作区命令调用 | 2 | 6 |
| 全部命令调用 | 3 | 7 |
| 耗时 | 39.133 秒 | 46.097 秒 |
| input tokens | 93,443 | 94,619 |
| cached input tokens | 59,392 | 52,224 |
| output tokens | 953 | 1,454 |
| reasoning output tokens | 163 | 300 |

token 数据来自 Codex CLI 的 `turn.completed.usage`。它包含运行时系统上下文和技能，不等于夹具本身的 Markdown token 数，因此只能描述本次真实执行成本，不能直接解释为记忆文件体积。

## 已验证的部分

- 两组都能在隔离的只读临时目录中完成任务。
- 原始 JSONL、最终回答、运行元数据和人工评分均可保留。
- 两组均能恢复当前目标、下一步、稳定规则、参考资料和状态分类。
- Baseline 与 Layered 使用相同提示和相同事实内容。

## 协议问题

### 1. 出现正确性天花板效应

夹具信息量太小，Baseline 的混合记录仍然很容易阅读。两组均为 10/10，继续重复当前协议很可能只能证明“两种方式都能完成小任务”，无法识别分层的收益或代价。

### 2. 综合问题破坏了按需加载测试

一个提示同时询问当前状态、稳定规则和参考资料，Layered 必然要读取全部三层。它不能验证“只在需要时读取某一层”，反而增加了导航调用。

### 3. 当前试跑不支持“读取次数更少”

Layered 的工作区命令调用为 6 次，Baseline 为 2 次。这个结果不能据此证明分层一定更慢，因为 Agent 的探索策略也产生了额外 `rg` 和 `AGENTS.md` 读取；但它至少否定了本次场景下“分层自然减少读取次数”的直觉。

### 4. 存在真实运行时噪声

两组都读取了全局 `using-superpowers` 技能，并出现技能描述预算提示。该调用没有提供任何夹具答案，对两组条件相同。正式分析必须分别报告工作区调用和运行时调用，不能把两者混在一起。

### 5. 模型需要显式锁定

试跑没有传入 `--model`，原始事件流也没有返回可直接核对的模型字段。当前本机默认配置指向 `gpt-5.6-sol`，但正式实验不能依赖隐式默认值，必须显式传入并记录模型。

## 协议修订方向

正式实验前进行第二版试跑：

1. 增加信息量和相似干扰项，但保持两组事实逐条等价。
2. 把综合问题拆成“恢复当前任务”“查询稳定规则”“查找参考资料”三个独立任务，每次使用新会话。
3. 为每类任务分别统计正确性、工作区读取、耗时和 token，检验按需加载是否成立。
4. 显式锁定同一模型，并交替执行 Baseline 与 Layered，减少顺序影响。
5. 增加结构维护任务，例如更新当前状态或替换失效结论，用文件差异验证分层是否降低误改范围。
6. 第二版试跑仍不计入正式样本；只有不再出现协议级歧义后，才运行每组至少 3 次。

## 当前能下与不能下的结论

可以下的结论：三层结构和混合记录都能在当前小型夹具中帮助新会话恢复完整信息；真实运行证据采集链路已经可用。

不能下的结论：三层结构比混合记录更准确、更快、更省 token 或读取更少。现有试跑没有支持这些判断。
