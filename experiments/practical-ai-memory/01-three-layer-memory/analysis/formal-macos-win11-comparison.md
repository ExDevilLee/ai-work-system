# macOS 与 Win11 正式实验对照报告

日期：2026-07-20

## 对照范围

本报告比较同一套三层记忆实验在 macOS 与原生 Windows 11 上的正式结果。
两个平台使用相同的冻结夹具、任务提示、标准答案和模型配置：

- 模型：`gpt-5.6-sol`。
- 推理强度：`medium`。
- 执行器：`codex-cli 0.144.1`。
- 沙箱：`read-only`。
- 样本：每个平台 3 个任务 × 2 个条件 × 3 次，共 18 次；合计 36 次。

本报告验证的是同一 POC 能否在两个平台独立复现，不验证 macOS 与 Win11
之间的记忆双向同步、冲突合并或恢复协议。

## 数据门禁

- macOS 18 次正式运行使用最新指标审计器重新审计，`18/18` 覆盖完整。
- Win11 18 次正式运行在原设备重新审计，`18/18` 覆盖完整。
- 两个平台均未重跑实验或重新评分。
- 两个平台 6 个任务/条件分组均为 `n=3`、`workspace_metrics_n=3`。
- Win11 的 4 个 workspace MCP 调用和 `9,488 B` 完整结果文本均被计量，未计量调用为 0。

Win11 聚合文件在传输时通过 SHA-256 校验。CSV 使用 LF 换行，仓库中的哈希保持不变；
JSON 从 Win11 的 CRLF 规范化为仓库约定的 LF，内容未改变：

- CSV：`908f6339c51e3bd3ec11ab695fb11353f80b15852717d2b06628d8ad3d7a5fe7`。
- JSON 原始传输文件：`8e3ab75398fbc406d0eb4e5768456a8927babd2808b673ff1242bb14d3ed3fac`。
- JSON 仓库 LF 文件：`e9e9330209d594fb92bdea6b47ed958dee18f9a22be979663f511dd02e99ae7d`。

## 正确性

| 任务 | macOS Baseline | macOS Layered | Win11 Baseline | Win11 Layered |
| --- | ---: | ---: | ---: | ---: |
| 当前任务恢复 | 15/15 | 15/15 | 15/15 | 15/15 |
| 稳定规则识别 | 30/30 | 30/30 | 30/30 | 30/30 |
| 参考资料检索 | 24/24 | 24/24 | 24/24 | 24/24 |

两个平台、两种结构均为满分。这支持“当前 POC 可以在两个平台完成相同任务”，
但仍不能证明 Layered 比 Baseline 更准确。

## 工作区文本读取

下表使用每组 3 次运行的中位数：

| 平台 | 任务 | Baseline | Layered | 减少比例 |
| --- | --- | ---: | ---: | ---: |
| macOS | 当前任务恢复 | 3,628 B | 575 B | 84.2% |
| macOS | 稳定规则识别 | 3,628 B | 1,595 B | 56.0% |
| macOS | 参考资料检索 | 4,058 B | 895 B | 77.9% |
| Win11 | 当前任务恢复 | 9,824 B | 2,600 B | 73.5% |
| Win11 | 稳定规则识别 | 10,055 B | 3,243 B | 67.7% |
| Win11 | 参考资料检索 | 6,232 B | 1,630 B | 73.8% |

两个平台的绝对读取量不同，但三个任务的方向一致：Layered 都读取了更少的工作区文本。
因此更稳妥的跨平台结论是“按职责分层在当前任务中稳定减少项目文本读取”，
而不是“某个平台固定会读取多少字节”。

| macOS | 稳定规则识别 | 1 | 2 | 38.504 秒 | 39.902 秒 |
| macOS | 参考资料检索 | 3 | 3 | 36.465 秒 | 40.552 秒 |
| Win11 | 当前任务恢复 | 4 | 5 | 70.157 秒 | 63.594 秒 |
| Win11 | 稳定规则识别 | 3 | 3 | 80.234 秒 | 62.594 秒 |
| Win11 | 参考资料检索 | 5 | 4 | 80.203 秒 | 66.188 秒 |

耗时在两个平台呈现相反方向：macOS 的 Layered 中位数略高，Win11 的 Layered
中位数更低。调用次数也没有统一变化。这说明延迟和调用数量明显受到工具选择、
运行时上下文、缓存和服务波动影响，不能归因于文件分层本身。

Win11 的 input token、output token 和绝对耗时整体高于 macOS，但当前实验没有隔离
所有运行时上下文和服务状态，因此这些数值只作为环境观察，不用于声称平台性能差异。

## Win11 暴露的兼容问题

Win11 验证过程中真实发现并修复了以下问题：

1. Git 自动换行可能把冻结夹具和提示从 LF 改为 CRLF，导致字节级哈希变化。
2. Windows 路径排序规则与 macOS 不同，必须按 POSIX 相对路径字符串排序。
3. Python 需要先解析实际的 `codex.cmd` 启动器，不能把裸 `codex` 直接交给 `CreateProcess`。
4. subprocess 输出必须显式使用 UTF-8，不能依赖 Windows 默认 GBK。
5. 多行中文提示必须通过 UTF-8 stdin 传给 Codex，避免 npm `.cmd` 参数转发截断。
6. MCP 返回可能使用 JSON 包装，审计器必须解码字符串叶节点后再匹配 fixture，
   同时仍按模型实际收到的完整结果文本计量字节数。

这些问题影响的是实验运行器和证据测量，不是三层 Markdown 内容本身。修复均有跨平台
回归测试，最终 36 次正式运行的冻结夹具、提示哈希和评分协议保持一致。

## 可以支持的结论

在当前固定模型、小型夹具和三个确定任务下：

- macOS 与 Win11 都能独立完成同一套三层记忆实验。
- 两种结构在两个平台都保持满分正确性。
- Layered 在两个平台、三个任务中都减少了工作区文本读取。
- 只用文件夹、Markdown 和工具规则入口即可形成可观察的按需加载行为。

## 不能支持的结论

- 不能声称分层一定提高正确性、减少调用或缩短耗时。
- 不能把两个平台的 token 和延迟差异解释为操作系统性能差异。
- 不能推广到所有模型、AI 工具、项目规模或真实长期记忆目录。
- 不能把“两个平台独立复现”写成“已经验证跨设备双向同步”。
