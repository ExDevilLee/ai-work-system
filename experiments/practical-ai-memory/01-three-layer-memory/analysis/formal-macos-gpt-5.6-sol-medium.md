# macOS 正式实验报告

日期：2026-07-17

## 实验配置

- 模型：`gpt-5.6-sol`。
- 推理强度：`medium`。
- 执行器：`codex-cli 0.144.1`。
- 平台：macOS 26.5，Apple Silicon。
- 会话：每次使用独立 `--ephemeral` 新会话。
- 沙箱：`read-only`。
- 样本：3 个任务 × 2 个条件 × 3 次，共 18 次预定义正式运行。

不同模型的数据没有并入本报告的正式样本。

## 正确性

| 任务 | Baseline | Layered |
| --- | ---: | ---: |
| 当前任务恢复 | 15/15 | 15/15 |
| 稳定规则识别 | 30/30 | 30/30 |
| 参考资料检索 | 24/24 | 24/24 |

两种结构在当前任务和夹具中均为满分，不能证明 Layered 比 Baseline 更准确。

| 任务 | 条件 | 工作区调用 | 工作区输出 | 耗时 | output tokens |
| --- | --- | ---: | ---: | ---: | ---: |

- 当前任务：约 84.2%。
- 稳定规则：约 56.0%。
- 参考资料：约 77.9%。

## 可以支持的结论

在本实验中，把当前状态、稳定规则和参考资料按职责分开，可以明显减少单一任务返回给 Agent 的项目文本，同时保持答案正确性。

这个收益来自更小、更明确的目标文件，而不是某种复杂记忆算法。只使用文件夹、Markdown 和工具特定入口说明，就已经能够形成可观察的按需加载行为。

## 不能支持的结论

### 分层不一定减少调用次数

稳定规则任务中，Layered 的调用中位数为 2，Baseline 为 1。参考资料任务两者中位数相同。索引和入口文件可能减少正文读取量，也可能增加导航步骤。

### 分层不一定更快

三个任务的 Layered 耗时中位数都没有低于 Baseline。延迟还受到模型服务、缓存和全局运行时行为影响，不能根据当前小样本声称分层会加速回答。

### 项目文本减少不等于计费 token 同比例减少

Codex CLI 的 input token 包含系统上下文、缓存和运行时技能。`workspace_output_bytes` 只测量工具返回的项目文本，两者不能直接换算。

### 结果不能直接推广到所有模型和项目规模

当前夹具较小，模型固定，任务答案确定。真实项目的文件数量、命名质量、入口说明和模型工具策略都会改变结果。

## 测量修正记录

一条正式运行把全局技能读取与工作区读取合并在同一条 shell 命令中。旧分类器曾把整条命令排除，导致工作区输出被低估。

原始 JSONL 被完整保留。修正后的 `audit_run_metrics.py` 只在聚合输出以已知全局文件正文为精确前缀时剥离该前缀；不能精确匹配时标记指标不可靠，不估算数值。全部 18 次主样本最终通过审计。

额外执行的 `formal-04` 配对用于验证修正算法，不并入预定义的 18 次主样本。
