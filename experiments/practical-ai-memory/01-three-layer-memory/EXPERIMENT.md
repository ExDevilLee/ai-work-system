# 实验协议

## 研究问题

只改变项目记忆的文件组织和加载入口，不改变信息内容，三层结构是否能提高新 AI 会话恢复任务、识别稳定规则和按需查找资料的准确性？

## 实验条件

### Baseline：混合记录

当前状态、稳定规则、参考资料索引、历史观察和失效信息放在同一个 `PROJECT_NOTES.md` 中。`AGENTS.md` 只指向这份混合记录。

### Layered：三层记忆

完全相同的信息按职责分开：

- `memory/CURRENT.md`：当前目标、进度、下一步和阻塞。
- `memory/MEMORY.md`：经过确认的稳定事实与规则。
- `references/INDEX.md`：按需资料入口，具体资料位于 `references/`。
- `AGENTS.md`：说明按任务类型加载上述文件。

两组不得增删事实、改变措辞强度或加入额外提示答案。允许为文件导航增加中性说明。

## 运行环境

- 已完成基线平台：macOS。
- 待验证平台：Windows 11，使用 `--platform-tag win11` 写入独立运行目录。
- 执行器：本机 Codex CLI，使用独立的 `--ephemeral` 会话。
- 沙箱：`read-only`。
- 每次运行使用从夹具复制得到的独立临时目录，避免运行间状态污染。
- 模型、Codex CLI 版本、命令参数和夹具校验和写入每次运行的元数据。

本轮正式实验固定以下模型配置：

- 模型：`gpt-5.6-sol`。
- 推理强度：`medium`。
- Codex CLI：每次运行实时记录版本。

Windows 使用相同协议延后验证，不阻塞当前 macOS 实验。真实 Win11 执行手册见 [`WINDOWS-VALIDATION.md`](WINDOWS-VALIDATION.md)。

## 测试任务

每次新会话使用同一个综合提示，要求 Agent 检查项目文件后回答：

1. 当前唯一目标是什么？
2. 下一步动作是什么？
3. 哪两条规则是已经确认、应长期遵守的稳定规则？
4. “Wiki 发布后检查”需要核对哪三项，并给出资料来源文件路径？
5. 哪一条信息只是待验证观察，哪一条旧结论已经失效？

提示不告诉 Agent 具体文件名或目录结构。

## 预期答案

标准答案维护在 `expected/answers.json`，评分时不得根据模型答案临时改变。

## 指标

| 指标 | 评分方式 |
| --- | --- |
| 当前目标正确 | 0 或 1 |
| 下一步动作正确 | 0 或 1 |
| 两条稳定规则 | 每条 0 或 1，共 2 分 |
| 三个发布后检查项 | 每项 0 或 1，共 3 分 |
| 来源路径正确 | 0 或 1 |
| 待验证观察识别 | 0 或 1 |
| 失效结论识别 | 0 或 1 |
| 无依据断言 | 计数，越少越好 |
| 文件读取与工具调用 | 从事件流统计；无法稳定识别时标记不可用 |
| 耗时与 token | 从执行器输出记录；缺失时标记不可用，不估造 |

正确性总分为 10 分。除总分外必须保留每项结果，避免总分掩盖不同失败类型。

## 试跑门禁

试跑用于验证协议，不用于支持文章结论。出现以下任一情况时，修订协议后重新试跑：

- 两组夹具的信息不等价。
- 提示存在多种合理解释。
- 标准答案无法进行确定评分。
- 执行器没有保留原始事件或最终回答。
- Agent 因权限或环境问题无法读取夹具。

试跑通过后，正式实验每组至少 3 次，并交替运行条件顺序，降低时间顺序造成的偏差。

### Pilot 01 结果

2026-07-17 的第一次双组试跑验证了运行与证据采集链路，但出现正确性天花板效应，且综合提示无法验证按需加载，因此协议尚未进入正式重复实验。具体数据和修订方向见 [`analysis/pilot-01.md`](analysis/pilot-01.md)。

### Pilot 02 协议修订

第二次试跑使用 `fixtures/pilot-02/`，并把综合问题拆成三个互相独立的新会话：

| 任务 | 目标 | Layered 预期读取 |
| --- | --- | --- |
| `current-task` | 恢复目标、下一步、阻塞和最近完成项 | `memory/CURRENT.md` |
| `stable-rules` | 查询稳定规则、待验证观察和失效结论 | `memory/MEMORY.md` |
| `reference-retrieval` | 查找 Wiki 清单和平台验证策略 | `references/INDEX.md` 及两个目标资料文件 |

Baseline 的相同事实混合在 `PROJECT_NOTES.md`，因此每个任务都必须加载同一份完整记录。Layered 按职责拆分，但事实内容和措辞强度必须保持一致。

Pilot 02 新增 `workspace_output_bytes` 指标：统计已识别的工作区命令或 MCP 工具返回给 Agent 的 UTF-8 字节数。该指标比单纯的调用次数更接近“实际加载了多少项目内容”，但仍不等于模型最终计费 token。

过程指标只统计当前解析器完整覆盖的读取方式。运行中如果出现尚未计量的 `mcp_tool_call`，正确性、耗时和 token 仍可保留，但工作区调用次数与输出字节必须标记为覆盖不完整，并从这两项的组内汇总中排除，不能记为 0。

Pilot 02 使用显式模型参数，六次运行按以下顺序交替执行，减少条件顺序影响：

1. `current-task`：Baseline、Layered。
2. `stable-rules`：Layered、Baseline。
3. `reference-retrieval`：Baseline、Layered。

第二次试跑仍只验证协议，不计入正式样本。三个任务都必须满足：答案可以确定评分、两组均无权限错误、原始证据完整、Layered 没有读取任务明确排除的层。通过后才进入每组至少 3 次的正式实验。

Pilot 02 首次执行 `reference-retrieval` 时发现 Baseline 只记录了原始资料路径，却没有实际创建对应文件，而 Layered 存在原始资料文件，两组不等价。该对运行保留为失败证据，不进入条件比较。修正后使用 `pilot-02b` 标签仅重跑参考资料任务。

## 模型敏感性对照

主实验先固定一个模型配置，避免把模型差异混入 Baseline 与 Layered 的结构比较。主实验完成后，再使用完全相同的冻结夹具、提示、沙箱和评分规则进行模型敏感性对照。

模型对照遵守以下规则：

- 每个模型或推理强度单独成组，不与 `gpt-5.6-sol medium` 合并计算。
- 至少比较正确性、失败模式、工作区输出字节、工作区调用和工具选择差异。
- 延迟和全局 token 记录为环境观察值，同时注明缓存和运行时噪声。
- 如果只完成单次对照，明确标记为探索性结果，不写成稳定模型排名。
- 公开文章在“实验环境”中列出主模型配置，在“模型差异说明”中解释其他模型复现时可能出现的行为差异。

## 数据保留

每次运行至少保存：

- `raw.jsonl`：执行器原始事件流。
- `final.md`：最终回答。
- `metadata.json`：运行时间、平台、工具版本、模型、参数、退出码、耗时和夹具校验和。
- `score.json`：逐项评分、评分依据和人工复核状态。

完整原始记录按平台写入被 Git 忽略的 `runs/private/<platform-tag>/`。完成评分后，由 `export_public_run.py` 生成同样按平台隔离的完整脱敏中间层 `runs/public/<platform-tag>/`，并执行敏感路径、会话 ID 和疑似密钥扫描。

提交到仓库的是 `build_committed_evidence.py` 生成的 `evidence/`：manifest 保留全部运行的评分、指标和校验和，代表目录只保留一轮完整正式对照、真实失败和模型差异样本，夹具与提示去重引用中央文件。

聚合数据写入 `data/`，分析和图示写入 `analysis/`。图示只有在正式样本足够、比较维度确实需要时才生成，并同时保留底层表格。

## 有效性限制

- 结果首先说明当前任务、当前夹具和当前模型下的表现，不能直接推广到所有 AI 工具。
- 文件数量、命名和导航说明本身可能影响结果，需要在文章中承认这是分层机制的一部分。
- 每组 3 次仍是小样本，只用于发现稳定趋势与失败模式，不作统计显著性宣称。
- Windows 结果完成前，不声称已经验证跨平台效果。
