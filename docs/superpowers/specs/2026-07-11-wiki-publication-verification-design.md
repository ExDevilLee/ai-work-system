# Wiki 发布前后自动检查设计

## 目标

为 GitHub Wiki 和 Gitee Wiki 增加可重复的发布验收，自动发现文章格式、配图路径、远端页面未更新和远端图片内容陈旧等问题，减少每次发布后逐页人工检查。

第一版不覆盖墨问，也不尝试判断配图是否美观、位置是否具有叙事效果；这些仍属于人工 Review。

## 方案

新增一个共享 Python 验证脚本，由现有两个 Wiki workflow 在发布前后分别调用。检查逻辑只依赖 Python 标准库、Git 和现有 Wiki 生成模块，不引入浏览器或截图比较。

### 发布前检查

- 使用标准 YAML 解析所有 `status: ready` 文章的 frontmatter。
- 检查正文中的本地配图存在、非空，并符合 `images/<两位文章编号>/<文件名>` 路径约定。
- 根据文件签名检查常用图片扩展名与实际格式一致。
- 在临时目录全量生成目标平台 Wiki。
- 检查 `Home.md`、`_Sidebar.md`、文章页数量、顺序和页面名。
- 检查每篇文章底部的上一篇、目录、下一篇导航及首尾“无”边界。
- 检查生成后的文章配图 URL 使用当前目标平台的 `asset_base_url`。

### 发布后检查

- 从目标平台重新拉取 Wiki Git 仓库，不复用发布时的本地 working copy。
- 根据同一批源文章重新生成预期 Wiki 文件，并逐页比较远端 Markdown 内容。
- 请求文章引用的每张远端图片，要求最终返回 HTTP `200`。
- 比较远端图片与仓库本地源图的 SHA-256，确认不是可访问但内容陈旧的旧图。
- 对远端仓库和图片的短暂可见性延迟执行有限次数、固定间隔重试；超过窗口后失败并输出平台、页面或图片路径。

## Workflow 接入

`Sync Wiki`：

1. 配置 GitHub Wiki 认证。
2. 运行 GitHub 参数的发布前检查。
3. 执行现有 Wiki 发布。
4. 运行 GitHub 参数的发布后检查。

`Sync to Gitee`：

1. 镜像主仓库到 Gitee。
2. 配置 Gitee Wiki 认证。
3. 运行 Gitee 参数的发布前检查。
4. 执行现有 Gitee Wiki 发布。
5. 运行 Gitee 参数的发布后检查。

任一检查失败都令当前 workflow 失败。墨问 workflow 的相关文件过滤保持不变，验证脚本和 Wiki workflow 的改动不应消耗墨问 API 配额。

## 错误输出

失败信息必须指出：

- 阶段：`pre-publish` 或 `post-publish`。
- 平台：GitHub Wiki 或 Gitee Wiki。
- 对象：文章源文件、生成页面、远端页面或图片 URL。
- 原因：格式不符、文件缺失、内容差异、HTTP 状态或哈希不匹配。

## 验证边界

自动检查能够证明源内容、生成结果、远端 Wiki 仓库和远端图片内容一致，但不能证明平台最终视觉排版一定美观。浏览器截图和视觉回归可作为后续增强，不进入第一版。
