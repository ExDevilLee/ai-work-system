# AI 长期记忆实战系列设计

日期：2026-07-16

状态：pending review

## 系列定位

第二系列暂定名称为“AI 长期记忆实战”，系列标识为 `practical-ai-memory`，计划用 9 篇文章讨论长期记忆的分层、边界、晋升、治理、检索、可视化、健康度和迁移。

系列核心表达：

> 长期记忆不是让 AI 记住更多，而是让不同信息进入正确的位置，并且能够被找到、更新、遗忘和恢复。

目标读者是已经感受到 AI 上下文断裂，希望在普通个人电脑上搭建可用长期记忆系统的读者。文章默认面向非专业开发者解释概念，但所有可操作结论都必须来自真实 POC，不用概念包装未经验证的方案。

## 不可妥协的原则

1. 证据先于文章。先完成 POC，再把经过验证的部分写进正文。
2. 失败也要保留。失败方案、环境限制和替代路径进入验证记录，不改写成顺利故事。
3. 不用单平台经验代替跨平台结论。macOS 可用不代表 Windows 可用。
4. 不把“理论上支持”写成“已经验证”。缺少执行证据时明确标记为待验证。
5. 不为了使用某种数据库、RAG 框架或知识图谱而制造需求。技术选型必须回应当前文章的问题。
6. 不要求读者默认使用 WSL、Docker 或云服务。它们可以作为可选路径，但基础方案应能在原生 macOS 和 Windows 环境运行。
7. 可照做的文章必须提供版本、安装、输入、命令、预期输出、失败排查和卸载或清理方式。

## 证据等级

每个 POC 使用以下状态：

| 状态 | 含义 | 能否写成实战结论 |
| --- | --- | --- |
| `proposed` | 只有问题、假设或候选技术 | 否 |
| `macos-verified` | 已在本机 macOS 完整执行 | 否，只能说明 macOS 结果 |
| `windows-ci-verified` | 已在 GitHub Actions `windows-latest` 完整执行 | 可以说明命令行和文件流程跨平台 |
| `windows-device-verified` | 已在真实 Windows 11 设备完成必要人工验证 | 可以说明 GUI、权限和桌面集成行为 |
| `article-ready` | 所需平台证据、复现说明和限制均已齐备 | 可以进入文章 `review` |

CLI、文件读写、数据转换、检索、备份和恢复流程，默认要求同时达到 `macos-verified` 与 `windows-ci-verified`。

涉及 GUI、系统权限、桌面应用、文件选择器、通知、系统服务或 Windows 特有路径行为时，除 CI 外还必须达到 `windows-device-verified`。无法获得真实设备证据时，对应内容不作为可照做的正式方案发布。

## Windows 验证组合

Windows 主验证环境使用 GitHub Actions 的 `windows-latest`：

- 使用全新 checkout，避免依赖开发机残留状态。
- 使用 PowerShell 执行 Windows 原生命令。
- 固定关键运行时和依赖版本。
- 执行与 macOS 等价的安装、初始化、样例、测试、导出和清理步骤。
- 失败时保留完整 job 日志；需要长期引用的结论提炼为仓库内验证报告。
- 对关键输出生成校验和或结构化摘要，避免只凭日志中的“成功”文字判断。

真实 Windows 11 设备只承担 CI 无法证明的部分，例如 GUI 显示、权限弹窗、桌面目录、系统托盘、长路径和安全软件影响。设备验证报告不保存用户隐私、主机名、密钥或个人目录明文。

## 技术选型门禁

候选技术进入 POC 前至少检查：

- macOS 与 Windows 是否都有官方或稳定安装路径。
- Windows 是否可以不依赖 WSL 完成基础流程。
- 项目维护状态、许可证和最近发布情况。
- 数据是否可以导出为开放格式，避免把长期记忆锁死在单一工具中。
- 是否可以通过脚本重复执行和自动验收。
- 是否能在普通 CPU、合理内存和本地磁盘条件下运行。
- 路径分隔符、编码、换行符、大小写和 Shell 差异是否有明确处理方式。
- 卸载、回滚、备份和恢复是否可行。

如果一个工具只在 macOS 表现良好，应把它标记为平台特定候选，而不是作为系列默认方案。若 Windows 需要明显不同的工具，文章可以提供两条经过验证的实现路径，但必须使用同一组输入与验收标准比较结果。

## POC 产物结构

每个主题验证后使用独立目录保存最小复现实验：

```text
experiments/practical-ai-memory/<topic>/
  README.md
  fixtures/
  scripts/
  tests/
  evidence/
    macos.md
    windows-ci.md
    windows-device.md
```

不是每个 POC 都必须拥有全部子目录。只保留复现所需内容，不提交大体积依赖、用户数据、密钥、临时缓存或可重新生成的原始日志。

POC 的 `README.md` 至少记录：

- 要验证的问题与反例。
- 候选方案及选择理由。
- 环境与版本。
- 最小输入和执行步骤。
- 自动验收标准。
- macOS 与 Windows 的结果。
- 已知限制、失败记录和清理步骤。
- 可以进入文章的结论，以及仍不能下的结论。

## 文章门禁

第二系列文章从大纲到发布采用以下顺序：

```text
问题与读者场景
  -> 候选方案与选型标准
  -> 最小 POC
  -> macOS 验证
  -> Windows CI 验证
  -> 必要时 Windows 11 设备验证
  -> 证据 Review
  -> 面向读者重写
  -> article review
  -> ready
```

文章进入 `status: review` 前必须满足：

- 正文中所有“可以”“会”“能够”“兼容”类实践结论都有验证证据。
- 命令和配置来自 POC 的最终通过版本，不从概念草稿复制。
- macOS 和 Windows 的差异单独列出，不让读者自行猜测。
- 至少有一个从初始化到验收的完整最小示例。
- 明确说明适用范围、已知限制和不推荐使用的情况。
- 配图来自已验证结构，不能画出实际实现中不存在的流程或组件。

## 九篇文章与验证对象

### 1. 工作记忆、长期记忆和可索引资料，为什么不能放在一起？

验证一个最小三层结构。使用同一组样例信息，证明临时任务状态、稳定事实和按需资料可以分别更新、加载和清理，不互相污染。

### 2. 哪些信息应该常驻，哪些应该按需检索？

验证上下文预算和按需装载策略。比较全部常驻与按任务检索在上下文体积、相关信息命中和错误带入方面的差异。

### 3. 个人长期记忆和项目长期记忆，为什么需要分开？

构造两个项目和一组个人偏好，验证作用域隔离、允许共享的最小信息，以及错误跨项目加载能造成的问题。

### 4. 一条临时经验，什么时候值得成为长期记忆？

验证从原始观察、候选结论、人工确认到长期记忆的晋升流程，并证明未确认观察不会自动成为稳定事实。

### 5. 当记忆冲突、过期或被替代时怎么办？

验证带来源、时间和状态的记忆治理流程，覆盖冲突发现、替代、降级、删除和历史追踪。

### 6. RAG 与 Memory 有什么区别？

使用同一份资料分别验证检索结果和长期记忆行为，比较“找到相关文本”与“维护稳定事实、状态和偏好”的差异。具体检索工具通过 POC 后再决定。

### 7. 为什么长期记忆需要可视化？

验证一个跨平台可访问的轻量查看界面或静态报告，确保读者能看到记忆来源、层级、更新时间和关联，而不是只能读取内部文件。

### 8. 怎么判断一套记忆系统是否健康？

验证覆盖范围、陈旧内容、冲突、孤立条目和待确认候选的检查方式，形成可以重复运行的健康报告。

### 9. 如何备份、恢复并迁移长期记忆？

验证从 macOS 导出、在 Windows 恢复，以及反向迁移的完整流程。使用校验和、条目数量和关键查询结果证明迁移前后一致。

## 技术选型节奏

不一次性为 9 篇文章锁定完整技术栈。每篇先定义与工具无关的验收标准，再选择最小候选完成 POC。

优先顺序：

1. 开放文件格式和系统自带能力。
2. Python、Node.js 等具备稳定 macOS 与 Windows 运行时的轻量脚本。
3. 可本地运行、可导出、维护活跃的成熟工具。
4. 只有在普通检索或关系表达不足时，才评估向量数据库、RAG 框架或 GraphRAG-lite。

任何后续更换工具的文章，都应保留相同验收样例，区分“概念成立”与“某个实现更合适”。

## 第一篇启动范围

第一篇先验证最小三层结构，不直接引入向量数据库、知识图谱或复杂 Agent 框架。

第一轮设计需要比较 2 至 3 种实现路径，并优先验证普通读者可理解、可手工检查、可在 macOS 和 Windows 原生运行的方案。只有 POC 通过后才创建文章初稿。

## 完成标准

- 9 篇文章都有独立、可追溯的 POC 或明确复用的共享证据。
- CLI 类实践同时有 macOS 与 Windows CI 结果。
- GUI 和系统集成类实践在需要时有 Windows 11 设备结果。
- 读者可以从干净环境按文章步骤得到与 POC 一致的结果。
- 未验证、失败或仅适用于单平台的内容均被明确标记。
- 系列文章不泄露本机路径、个人信息、密钥或私有记忆内容。
