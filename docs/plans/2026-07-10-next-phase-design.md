# AI Work System 下一阶段设计

## 目标

前四篇文章已经完成从长期 AI 工作系统的基本主张，到最小项目记忆系统的认知与实践闭环。下一阶段不再重复解释“为什么需要记忆”，而是把项目从文章集合推进为同时包含文章、模板和可复用方法的实践仓库。

这一阶段按以下顺序推进：

1. 创建最小项目记忆模板。
2. 用模板制作过程中的真实判断完成第五篇文章。
3. 整理选题池和 README，让后续主线从“记忆”进入“规则与工作流”。

## 最小项目记忆模板

模板放在 `templates/minimum-project-memory/`，采用“使用说明 + 可直接复制的项目目录”结构：

```text
templates/minimum-project-memory/
├── README.md
└── project/
    ├── README.md
    ├── AGENTS.md
    └── PROJECT-STATE.md
```

各文件职责：

- 模板根目录 `README.md`：解释适用场景、使用步骤、文件职责和维护边界。
- `project/README.md`：给人和 AI 看的项目入口，说明目标、范围、内容源头和常用入口。
- `project/AGENTS.md`：给 AI 看的协作规则入口，约定读取顺序、修改边界、验证方式和敏感信息边界。
- `project/PROJECT-STATE.md`：记录当前目标、已完成事项、下一步、阻塞和最近决策。

模板只提供最小骨架，不预置向量数据库、检索增强生成（RAG）、自动摘要、复杂任务系统或跨工具同步。所有占位内容都应能被首次使用者直接理解和替换。

## 第五篇文章

第五篇标题：

> 从聊天记录到项目规则：一次经验什么时候值得沉淀？

文章直接承接第四篇结尾的问题：“这件事以后还会重复吗？如果会，它应该沉淀到哪里？”

核心判断不是“重要的都记下来”，而是一次经验需要依次回答：

1. 它是否来自真实发生的问题？
2. 判断是否经过验证，而不只是临时猜测？
3. 未来是否可能重复影响工作？
4. 它应该停留在观察记录，还是晋升为规则、工作流或脚本？

文章使用项目中的真实案例：

- 系列文章钩子冲突，最终晋升为 `AGENTS.md` 规则。
- Gitee Wiki 手动拖拽排序属于平台隐藏状态，不晋升为可靠规则。
- `status: ready` 适合成为结构化发布状态。
- README 索引容易遗漏，因此从人工提醒晋升为脚本和自动化检查。

文章结尾自然引向下一阶段：当一条规则开始反复执行时，它什么时候应该继续变成工作流？

- 模板中的三个项目文件职责不重叠，复制后可以直接开始填写。

- 第五篇遵守“主题构思 -> 大纲设计 -> 初稿 -> 打磨”的推进顺序。
- 第五篇不重复第四篇已经展开的最小文件清单，而是讨论经验晋升标准。
- 第五篇与第四篇结尾、下一篇“从提示词到工作流”形成连续钩子。
- 文章进入 `status: ready` 后，README、多语言入口和 Wiki dry-run 均通过项目现有脚本验证。
