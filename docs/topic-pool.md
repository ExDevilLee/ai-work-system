## 近期优先

- 最小可用项目记忆系统：哪些文件先出现，哪些规则先写下来。
  - 承接第二篇末尾留下的问题。
  - 可以从 `README`、`AGENTS.md`、发布规则、文章状态和本地工作日志这些最小文件讲起。

- 为什么 AI 协作需要一份给 Agent 看的 README。
  - 可以解释 `AGENTS.md`、`CLAUDE.md` 这类 agent rule 文件的作用。
  - 重点不是文件名，而是“给 AI 看的项目入口”和“给人看的项目入口”有什么不同。

- AI 不是记不住，而是不知道该相信哪份上下文。
  - 适合展开 source of truth、上下文优先级和过期信息问题。
  - 可以用 README、Wiki、源稿、历史对话、记忆文件并存时的冲突来举例。

## 系统边界

- 长期 AI 工作系统里，哪些东西不应该被记住。
  - 可以讨论隐私、临时判断、过期状态、平台隐藏状态和不可验证信息。
  - 重点是“记忆不是越多越好”。

- 从聊天记录到项目规则：一次经验什么时候值得沉淀。
  - 可以用“系列文章要回看上一篇钩子”作为值得沉淀的例子。
  - 也可以用 Gitee 手动拖拽顺序作为不适合依赖隐藏状态的例子。

- 为什么长期工作系统需要清理，而不是只需要积累。
  - 可以讨论旧页面、旧规则、旧 worklog、过期任务和重复说明如何影响 AI 判断。

## 人的角色

- AI 工作系统为什么需要 Review，而不是只需要生成。
  - 可以承接文章审美、README 入口、Gitee 展示、发布链路这些真实 review。
  - 重点是人的价值从“亲自写所有东西”转向“判断什么值得写、什么算对”。

- 为什么自动化发布之后，人反而更要保留最后判断。
  - 可以从 GitHub Wiki / Gitee Wiki / CI 同步链路讲起。
  - 自动化减少重复执行，但不替代体验检查和方向判断。

- 当 AI 成为长期协作者，人的工作方式会怎么变。
  - 适合作为阶段性总结篇。
  - 可以呼应 README 里的核心主题：真正改变的不是效率，而是工作方式。

## 实践展开

- 一个不会从零开始的 AI 助手，实际能帮我做什么。
  - 可以用写作、发布、README 修正、CI 排障、跨平台展示这些连续任务举例。

- 从 prompt 到 workflow：如何把 AI 变成个人工作系统。
  - 可以解释从一次性指令到可复用流程、脚本、规则和验证链路的变化。

- 为什么公开仓库适合承载一部分 AI 工作系统。
  - 可以讨论公开内容、脱敏案例、发布流水线和私有上下文之间的边界。

- 多语言 README 够不够，文章要不要全部翻译。
  - 可以沉淀当前策略：入口国际化，正文中文优先，完整译文按需维护。

## 维护规则

- 新增选题时，优先记录它来自哪次真实实践或哪篇文章留下的问题。
- 写新篇章前，先回顾上一篇文章末尾的钩子，确认新篇是否承接、偏离或需要桥接。
- 选题库只是备选材料，不直接决定发布顺序。
- 如果某个选题已经写成文章，应从“近期优先”移走，或补充已发布文章链接。
