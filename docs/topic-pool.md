## 系统边界

- 为什么长期工作系统需要清理，而不是只需要积累。
  - 可以讨论旧页面、旧规则、旧工作日志（worklog）、过期任务和重复说明如何影响 AI 判断。

## 人的角色

- 当 AI 成为长期协作者，人的工作方式会怎么变。
  - 适合作为阶段性总结篇。
  - 可以呼应 README 里的核心主题：真正改变的不是效率，而是工作方式。

## 实践展开

- 一个不会从零开始的 AI 助手，实际能帮我做什么。
  - 可以用写作、发布、README 修正、CI 排障、跨平台展示这些连续任务举例。

- 为什么公开仓库适合承载一部分 AI 工作系统。
  - 可以讨论公开内容、脱敏案例、发布流水线和私有上下文之间的边界。

- 多语言 README 够不够，文章要不要全部翻译。
  - 可以沉淀当前策略：入口国际化，正文中文优先，完整译文按需维护。

- 一张正文配图如何同时发布到 GitHub Wiki、Gitee Wiki 和墨问。
  - 可以从本地相对路径、展示层地址改写、墨问图片 UUID 和三平台验证过程展开。
  - 重点不是工具配置，而是同一内容源如何适配不同平台的图片协议。

- 为什么新增一篇文章，还需要重新生成上一篇文章。
  - 来自 GitHub Wiki 和 Gitee Wiki 连续阅读导航的实际改造：读者看完第 n 篇后，不应该先记住标题、返回目录树，再寻找第 n+1 篇。
  - 可以讨论“源文不变、展示层全量重建”的边界，以及新增第 9 篇时为什么必须同步更新第 8 篇的下一篇链接。
  - 重点不是三个链接本身，而是自动化生成内容时，变更影响范围往往不只包含新增文件。

- 流水线显示成功，为什么还需要发布后的自动检查。
  - 来自正文配图发布后的真实验证过程：生成成功、仓库推送成功，不等于远端页面已经使用最新 Markdown，也不等于图片 URL 已经可访问且内容正确。
  - 可以串联配图发布、Gitee Raw 延迟、连续阅读导航和发布后 smoke test，展示系统如何在一次次人工检查中识别出可自动化的判断。
  - 重点是 Review 分工：格式、路径、HTTP 状态和哈希一致性交给脚本；配图是否美观、位置是否自然仍留给人。

## 维护规则

- 新增选题时，优先记录它来自哪次真实实践或哪篇文章留下的问题。
- 写新篇章前，先回顾上一篇文章末尾的钩子，确认新篇是否承接、偏离或需要桥接。
- 选题库只是备选材料，不直接决定发布顺序。
- 如果某个选题已经写成文章，应从“近期优先”移走，或补充已发布文章链接。
