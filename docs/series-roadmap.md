# AI Work System 系列写作路线图

本文记录系列级写作方向、暂定篇数和主题顺序。

它是一份可调整的 Roadmap，不是发布承诺。具体文章仍需结合已有系列骨架、上一篇结尾钩子和当时发生的真实实践决定。

文档职责：

- 本文负责系列方向、篇数、阶段和先后顺序。
- [`topic-pool.md`](topic-pool.md) 负责收集尚未排期的单篇候选与实践素材。
- README 和 Wiki 负责展示已经进入 `ready` 状态的正式文章。

## 总体路线

当前暂定四个系列：

| 顺序 | 系列 | 暂定规模 | 当前状态 |

## 系列一：长期 AI 工作系统

系列标识：`long-term-ai-work-system`

目标读者：希望从一次性 AI 对话走向长期协作的普通用户、创作者和个人项目维护者。

核心问题：如何让 AI 协作拥有记忆、流程、证据、Review、恢复能力和清晰的人机边界。

### 当前进度

第 1 至 15 篇已经进入 `ready`，当前系列已经完成。正式标题和阅读入口以 README 的“系列文章”为准。

已经覆盖的主线：

1. 从一次性聊天转向长期工作系统。
2. 解释 AI 失忆、外部记忆和最小项目记忆结构。
3. 从经验沉淀推进到规则和工作流。
4. 建立 Review、证据与人机决策边界。
5. 讨论人的角色、系统价值、复杂度和清理机制。
6. 用完整写作与发布过程说明任务连续性。

### 系列收束情况

- **13. 有了长期工作系统，还需要每次给 AI 写很长的提示词吗？**（已进入 `ready`）
  - 讨论外部上下文建立后，提示词从“携带全部背景”转向“说明当前意图和变化”。
- **14. 哪些工作值得进入长期 AI 系统，哪些只适合临时聊一次？**（已进入 `ready`）
  - 讨论任务频率、连续性、风险和维护成本，避免把所有对话都系统化。
- **15. 当 AI 越来越了解你的工作，哪些信息不应该进入系统？**（已进入 `ready`）
  - 讨论隐私、敏感信息、公开边界、最小必要上下文和不可自动代理的内容，作为当前系列收束。

系列一已在第 15 篇完成。若新的内容只是重复解释记忆、规则、工作流或 Review，应优先并入已有文章；只有出现新的核心问题和足够真实证据时，才考虑补充或调整系列内容。

## 系列二：AI 长期记忆实战

系列标识暂定：`practical-ai-memory`

暂定规模：8 至 10 篇。

目标读者：已经感受到 AI 上下文断裂，希望进一步理解和实践长期记忆的人。

这个系列不把长期记忆等同于“保存全部聊天记录”，也不一开始引入复杂数据库。重点解释记忆如何分层、检索、晋升、治理、可视化和恢复。

### 实践与验证约束

第二系列采用“先验证 POC，再写公开文章”的方式推进，不先写概念稿再补实践。这里的 POC 指用最小可行方案验证关键判断的原型实验。

- 每篇实战文章必须对应已经运行过的本地实验、真实案例或可复查证据；只有概念推演时，继续留在选题或实验阶段。
- 技术选型同时考虑 macOS 和 Windows，但采用分阶段验证：macOS 是当前主体实验和首版文章的发布门槛；Windows 兼容性验证随后补充，不阻塞已经在 macOS 获得充分证据的实验与写作进度。
- 第一阶段优先采用文件夹和 Markdown 作为基础层，自动化脚本再按需要引入 Python 3.12；不为了展示技术复杂度提前加入向量数据库、RAG 或 GraphRAG。
- 每个 POC 先定义场景、输入、预期结果、失败条件和跨平台验收证据，再决定它能支持哪一篇文章。
- 每个 POC 保留完整实验过程、环境元数据、原始输出、失败记录和派生数据；公开文章中的测试数据必须能够追溯到这些证据。
- 每次 LLM 实验显式记录模型名称、推理强度和执行器版本。不同模型或推理强度的数据分组报告，不混合成同一组样本；文章说明当前结论在哪个模型配置下得到，并单列模型敏感性对照，避免读者把模型差异误解为方案无法复现。
- 每个阶段性 POC 完成后，按 [`poc-evidence-publication.md`](poc-evidence-publication.md) 把可复现部分脱敏并提交到仓库的 `experiments/` 目录。公开 POC 使用全量 manifest、少量代表样本和去重夹具；密钥、用户数据、本机绝对路径、会话标识和不适合公开的原始运行上下文只保留在本地私有证据区。
- 第二系列文章增加“实验与复现”章节，链接到支撑当前文章结论的 GitHub POC 目录；读者应能看见实验如何执行，并能用公开夹具快速复跑，而不是只能接受正文结论。
- 图示按证据表达需要选择。对比维度较多且图表能明显降低理解成本时，可以加入表格、柱状图、雷达图或其他合适图示；不为了形式统一强制配图。
- 第二系列每篇文章末尾设置“参考文献”，优先引用论文、标准、官方文档和原始项目资料，并采用统一的学术文献格式。引用关系在实验阶段同步记录，不在成稿后临时补造来源。
- 公开文章区分已经验证的结论、仍在试用的机制和后续候选，不把单次成功包装成普遍规律。

第一篇《只用文件夹和 Markdown，怎么搭一个最小可用的三层 AI 记忆系统？》已经发布。对应 POC 在 macOS 与原生 Win11 上各完成 18 次正式运行，共 36 次；另有 6 次 macOS 探索性模型对照。公开文章、聚合数据、代表样本和复现脚本均已进入仓库，私有原始事件继续保留在各自设备的本地证据区。

第二篇《长期记忆里，哪些信息应该常驻，哪些应该按需读取？》已完成 macOS 与原生 Win11 各 36 次正式运行，共 72 次，并根据跨平台结果发布。两个平台共同支持按信息类型设计加载路径，但 Win11 的沙箱拒绝与重试也证明按需读取的实际成本受 Agent 工具链影响。

第三篇《一条临时经验，什么时候值得成为长期记忆？》已进入 `review`。对应 POC 已完成 macOS 45 次、Win11 45 次正式运行；两个平台都观察到直接晋升出现 6 次误晋升，规则门禁和分阶段人工门禁未出现误晋升。Win11 已完成真实计时评分和脱敏聚合，文章新增跨平台结果后仍等待人工 Review，再推进到 `ready`。

### 内容目录规则

- 第二系列文章统一放在 `content/articles/practical-ai-memory/`，frontmatter 的 `series` 使用 `practical-ai-memory`，目录与字段必须一致。
- 正文图片放在 `content/articles/practical-ai-memory/images/<系列内文章编号>/`，第一篇从 `01` 开始；正文使用 `images/01/<文件名>` 相对路径。
- 文章编号、上一篇和下一篇都只在第二系列内部计算，不延续第一系列的 `01` 至 `15`。
- 每篇文章的“实验与复现”链接指向对应 `experiments/practical-ai-memory/<POC>/`，不链接本地私有运行目录。

### 候选主题骨架

1. 工作记忆、长期记忆和可索引资料为什么要分层。
2. 哪些信息应该常驻，哪些应该按需检索。
3. 一条经验如何从原始记录晋升为长期记忆。
4. 如何处理记忆冲突、过期、替代和降级。
5. RAG 与 Memory 的区别：找到信息不等于形成记忆。
6. 为什么长期记忆需要可视化，而不应只藏在文件和索引里。
7. 如何查看记忆覆盖范围、活跃度和待治理内容。
8. 长期记忆如何备份、恢复并迁移到其他 AI 工具。
9. 事实之外的程序性记忆：规则、Skill 和工作流怎样被系统长期复用。

可选扩展：

- 轻量关系图和 GraphRAG-lite 应该在什么情况下出现。
- 个人长期记忆与项目长期记忆为什么需要分开。

### 新增实践分享候选

下面两个方向来自已经落地或正在验证的真实机制，先作为 Roadmap 候选记录。具体写作时仍需重新设计公开 POC、补齐跨平台证据，并根据当时的系列骨架决定篇号和位置。

1. **实践分享：长期记忆如何备份、恢复并跨平台迁移。**
   - 对应候选主题骨架中的“长期记忆如何备份、恢复并迁移到其他 AI 工具”，后续重点补充真实的 macOS 与 Win11 操作链路。
   - 文章可以讨论备份边界、版本与校验、增量同步、恢复门禁、冲突处理、失败回滚，以及 Git、归档包、对象存储等技术栈选项。
   - 可以参考当前已经落地的记忆备份与恢复机制设计实验，但不得直接公开真实记忆、账号信息、密钥、设备路径、会话标识、存储地址或未脱敏的运行记录。

2. **实践分享：跨平台任务如何在多个 AI 会话之间继续协作。**
   - 讨论 macOS 与 Win11 上的任务如何建立受控绑定，并通过本地 Bridge（桥接程序）和 Relay（中继控制服务）传递状态、指令、回执与停止信号。
   - 重点解释职责划分、会话绑定、确认与自动模式、幂等、重试、暂停、撤销、审计和失败恢复；不把某个 Agent 工具的私有会话实现描述为通用协议。
   - 文章只分享经过隔离 POC 验证的架构思路、协议边界、技术栈选择和脱敏证据，不公开真实会话 ID、用户信息、凭据、内网地址、对象存储签名链接或私有任务内容。

第二个方向同时涉及任务执行闭环。若最终内容更侧重任务编排、状态推进和跨设备恢复，应作为第二系列到 Loop Engineering 的桥接实践，而不是为了凑篇数强行放入长期记忆主线。

其中“程序性记忆”不能只根据外部研究中的 Skill 采用率直接得出效果结论。进入公开文章前，应通过重复任务 POC，对比临时提示词与稳定 Skill / Guide 在成功率、重复说明量、工作区读取量、返工和人工 Review 时间上的差异。

### 启动条件

- 系列一已经完成或只剩收尾工作。
- 至少准备好前三篇的连续骨架和第一篇的真实案例。
- 第一篇对应的三层记忆 POC 已在 macOS 完成重复实验，并保留可复查的输入、原始输出、实验数据和失败记录。
- Windows 验证已有明确的后续验收路径；未完成前，文章不声称方案已经在 Windows 实测通过。
- 面向普通读者解释概念，不直接复制 CodexClaw 的内部实现文档。

## 系列三：Loop Engineering：AI 工程闭环实战

系列标识暂定：`loop-engineering`

暂定规模：8 至 10 篇。

目标读者：希望让 AI 从“一次生成”走向可计划、可验证、可恢复交付的个人开发者、技术人员和 Agent 使用者。

这里的 Loop Engineering 指 CodexClaw 正在验证的 Agentic Engineering Loop（AI 工程闭环）实践，不把它描述为已经形成统一标准的业界方法论。系列重点是把一次 AI 任务组织成持续闭环：从原始需求归档和 Spec 分解开始，经过前置业务决策、交付计划、实现、验证和人工验收，再把结果与经验回写到 Spec、流程和能力候选中。

### 候选主题骨架

1. AI 生成越来越快，为什么交付没有同样变快。
2. 为什么需求进入 AI 流程前，要先保存一份可追溯的原始文档。
3. 从 Epic、User Story、AC 到 Task，怎样分解才不会逐层偏离原始目标。
4. 对抗性检查如何在写代码前暴露需求、角色、权限和状态冲突。
5. Human Gate 应该放在哪里，才能前置关键决策又不成为交付卡点。
6. Delivery Plan 如何表达依赖、停止条件、风险和验证路径。
7. 每个阶段凭什么进入下一步：Evidence Pack 与 Gate Receipt。
8. 代码实现、自动检查、测试和 Fix Bug 如何形成机器可继续的循环。
9. 人工验收失败后，为什么不应该默认只回到代码层修 Bug。
10. 一次交付结束后，如何回写 Spec、流程和 Skill 候选，形成下一轮更好的闭环。

可选扩展：

- 长任务中断后，如何从 checkpoint（检查点）恢复而不是全部重来。

### 优先验证候选

- 什么时候应该拆分多个 Agent，什么时候单线程更好。
  - 分别选择任务可以独立验收和必须共享状态的场景，对比单 Agent 与并行 Agent 的质量、时间、冲突成本和人工协调成本。
  - 在形成对照证据前，不用 Agent 数量、并发数或累计运行时间代表交付效率。

### 与 Team SDD 的边界

- Loop Engineering 关注单个任务或一组执行任务如何完成闭环：计划、执行路径、验证、人工决策、停止、恢复和经验回收。
- Team SDD 关注团队如何共享和治理项目认知：Spec、角色、Review Slice、追踪关系、变更边界和团队决策权。
- Evidence Pack 和 Human Decision Gate 会在两个系列中出现。Loop Engineering 先解释它们在任务闭环中的作用，Team SDD 再讨论它们如何进入多人协作协议，避免重复写成两套概念。

### 启动条件

- 长期记忆系列已经提供足够的状态、恢复和经验沉淀基础。
- 至少准备好一个完整真实任务，能够展示 `Requirement -> Spec -> Plan -> Execute -> Verify -> Accept -> Reconcile` 全过程。
- 明确区分已经通过真实任务验证的做法、仍在试用的机制和仅供参考的外部观点。
- 不为了展示多 Agent 而强行并行；简单任务仍保留单线程路径。

## 系列四：AI 时代的 Team SDD

系列标识暂定：`team-sdd`

暂定规模：8 至 10 篇。

目标读者：正在将 AI 引入软件研发流程的开发者、技术负责人和协作团队。

这里的 SDD 指 Spec-Driven Development（规范驱动开发）。系列重点不是增加文档，而是把人的意图、约束、验收、风险和不可代理判断，变成 AI 可以执行、检查和追踪的协作契约。它建立在前一系列的任务闭环基础上，但把讨论范围扩展到多人、多角色和长期团队治理。

### 候选主题骨架

1. 为什么 AI 写代码越快，团队 Review 反而可能越慢。
2. Spec 为什么是认知契约，而不只是需求文档。
3. 产品、架构、开发、测试等角色如何共同维护项目认知。
4. 什么是 Review Slice，为什么不应默认等同于拆分 PR。
5. Evidence Pack 如何减少人工重复验证。
6. 哪些判断应该进入 Human Decision Gate（人工决策关口）。
7. 新会话、上下文压缩和多人协作后，项目如何继续推进。
8. Runtime Hooks 如何在执行过程中约束 AI，而不只依赖提示词。

可选扩展：

- 老项目如何通过 `As-Is / To-Be / Gap` 接入 SDD。
- 小团队如何采用最小 Team SDD，而不是建设文档工厂。

后续实验和文章还应关注同一模型在不同团队环境中的差异。上下文、权限、培训、Review 机制和治理配套都可能影响最终结果，不能把 Agent 数量、token 或 Skill 调用率当作团队成熟度。Team SDD 更适合评价可追踪决策、Evidence Pack 完整度、人工决策成本和错误逃逸。

## 跨系列证据框架

后续 POC、文章和复盘优先从五个维度判断一个机制是否真正有效：

1. `Adoption`：是否在真实任务中再次使用，而不只是完成一次演示。
2. `Execution`：是否形成实际工件，并通过约定的验证和验收。
3. `Reuse`：是否减少重复上下文、重复读取或重复人工说明。
4. `Quality`：返工、遗漏、错误逃逸和 Review 发现的问题是否减少。
5. `Human cost`：人工审查时间、必须中断的决策点和总 lead time 是否可控。

这五个维度是跨系列的观察框架，不要求每篇文章机械展示全部指标。token、运行时间、并发数和 Skill 调用率可以作为过程数据，但不能单独代表质量、业务价值或生产率。

### 启动条件

- 长期记忆和 Loop Engineering 系列已经建立足够的上下文、证据与任务闭环概念，或者 Team SDD 出现更强的真实团队实践信号。
- 每篇文章都能对应一个真实协作痛点、案例或验证证据。
- 专业术语提供中文解释，避免只面向熟悉 SDD 工具生态的读者。

## 穿插型实践文章

下面这些主题可以作为系列中的案例，也可以暂时保留在选题池，不必单独扩展成大型系列：

- 同一张正文配图如何发布到多个平台。
- 为什么新增一篇文章还要重新生成上一篇导航。
- 流水线成功后为什么仍然需要远端自动检查。
- 多语言入口和中文正文如何划分维护边界。
- 公开仓库适合承载哪些 AI 工作系统内容。

是否插入主系列，要看它能否帮助解释一个更普遍的问题，而不只是记录平台操作步骤。

## 维护规则

- 每完成一篇文章，更新当前系列进度和下一篇暂定方向。
