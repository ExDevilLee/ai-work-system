# Agent Instructions

- 遇到有文章新增或修改时，默认运行 `python3 scripts/update_readme_index.py`，同步更新 `README.md` 和所有 `README-xx.md` 多语言版本。
- 新建 CodexClaw 分享类文章时，按阶段逐个推进：主题构思 -> 大纲设计 -> 初稿 -> 打磨。每完成一个阶段后，用简短语言汇总三点：当前阶段做了什么；相比较上个阶段的改进；下一步计划怎么做。
- 文章进入 `status: review` 前，必须完成配图必要性判断；如果流程、层级、对比、状态变化或概念关系用纯文字不易理解，应先生成简单、面向非程序员的解释图，插入正文并完成本地预览；如果配图不能明显降低理解成本，则明确说明本篇无需配图，不为统一形式强行添加。
- 系列文章写新篇章时，先回顾上一篇文章末尾留下的钩子、预告或开放问题，确认新篇主题是否承接、偏离或需要桥接；如果存在冲突，优先修正上一篇结尾或在新篇开头补过渡说明。
- 在主题构思或大纲阶段，同时提供 2 至 4 个下一篇主题候选，简要说明各自与当前文章的承接关系、受众范围和可能重复的既有内容，由用户选择或确认推荐方向；初稿定稿前锁定结尾钩子，避免文章进入 `status: review` 后才发现下一篇主题过窄或与系列主线不符。
- 文章中连续列举三项及以上并列的问题、结论或建议时，优先使用 Markdown 列表逐项分行，避免挤在同一段中影响阅读；确有连续叙事需要时可以例外。
- 文章元信息（frontmatter）的字符串如果包含英文半角冒号 `:` 等 YAML 特殊字符，必须使用引号包裹；文章进入 `ready` 前要确认 frontmatter 能被标准 YAML 解析器读取，不能只依赖仓库脚本的简化解析结果。
- 本地推进步骤可以记录到 `WORKLOG.md`，供新会话参考；该文件只作为本地工作日志，不提交。文件过大时，按时间线清理旧记录，保留最近和仍有行动价值的内容。
- `content/articles/*.md` 是唯一内容源头；GitHub Wiki、个人网站、公众号等都只是展示层或分发层，不反向修改源文。
- GitHub Wiki 只同步 `status: ready` 的文章；已发布文章如果主仓库源文更新，重新运行同步脚本覆盖更新 Wiki。
- 墨问也只同步 `status: ready` 的文章；`publishing/mowen-notes.json` 是源文件与墨问笔记 ID 的持久映射，新文章发布后必须按时间倒序重建《AI 长期工作系统》目录笔记。
