# 从聊天记录到项目规则：一次经验什么时候值得沉淀？

上一篇文章最后，我留下了一个问题：

> 这件事以后还会重复吗？如果会，它应该沉淀到哪里？

写完那篇文章以后，我真的做了一套[最小项目记忆模板](https://github.com/ExDevilLee/ai-work-system/tree/main/templates/minimum-project-memory)。里面只有三个项目文件：`README.md`、`AGENTS.md` 和 `PROJECT-STATE.md`，分别回答“项目是什么”“AI 应该怎么做”“现在做到哪里”。

文件有了，下一个问题马上就出现了：什么内容有资格写进去？

如果每次讨论、每个想法、每次页面变化都记下来，这三个文件很快就会变成另一堆需要筛选的聊天记录。AI 虽然读到了更多文字，却更难判断哪些仍然有效，哪些只是当时的猜测。

所以，沉淀不是简单地按下保存键。

它更像一套筛选过程：一段对话先变成观察，观察经过验证后才可能成为经验，反复影响工作的经验才值得进入项目规则；只有那些稳定、机械、可以检查的规则，才适合继续变成工作流或脚本。

## 1. 不是每段有用的对话都应该变成长期规则

一段对话当下有用，不等于它长期有效。

比如我们讨论一篇文章的标题，列出十几个候选，最后选定其中一个。那些候选帮助了当时的判断，但没有必要全部写进项目规则。真正值得留下的，可能只是“文章标题要和已发布名称保持一致，避免读者误以为是两篇内容”。

又比如某个平台今天的页面布局发生了变化。把它记在排障记录里也许有用，但如果没有确认它是否稳定、是否由项目控制，就不应该立刻要求后续所有 AI 都按这个现象工作。

未经筛选的沉淀，会带来三个问题。

1. 临时信息会伪装成长期事实。
2. 不同时间留下的结论可能互相冲突。
3. 规则越来越多，却没有人知道什么时候应该删除。

最后，AI 每次进入项目时虽然读了很多，却要重新猜哪一条才算数。

这和没有记忆是两种相反的失控：一种是什么都没有留下，另一种是什么都不肯丢掉。

## 2. 一次经验先经过四个问题

我现在更愿意先问四个问题。

第一个问题是：它来自真实发生的问题，还是我想象中的问题？

真实问题通常有明确的触发点。可能是文章顺序真的乱了，README 状态真的过期了，或者一次发布真的漏掉了某个步骤。相比之下，“以后也许会需要”很容易让人提前设计一套从未被使用的复杂规则。

第二个问题是：这个判断经过验证了吗？

一次页面异常可能来自缓存，一次命令失败可能来自临时网络，一次 AI 回答跑偏也可能只是当前上下文不足。没有复现、对比或结果检查之前，它更适合留在观察层，而不是马上变成长期约束。

第三个问题是：它未来还会重复影响工作吗？

只出现一次、影响很小、重新判断成本也很低的事情，不一定值得维护一条规则。相反，如果同一个问题会在每篇文章、每次发布或每个新会话里出现，沉淀的价值就会快速增加。

第四个问题是：它应该停在哪一层？

有些信息只需要记录当前状态；有些需要成为协作规则；有些已经形成固定步骤；还有少数动作足够稳定，可以交给脚本执行。选择合适的层级，比单纯决定“记不记”更重要。

这四个问题不是一套正式审批流程。它们只是提醒我：不要把“刚刚讨论过”直接等同于“以后都应该这样做”。

## 3. 从观察到脚本，不是一步跳过去

如果要把这个过程画成一条路径，大概是这样：

```text
临时观察 -> 已验证经验 -> 项目规则 -> 工作流 -> 自动化脚本
```

我把从左向右的变化称为“晋升”。它不是说右边一定更高级，而是表示一条信息正在获得更长的有效期、更大的影响范围，也需要更强的证据。

临时观察只需要说明“刚刚发生了什么”。它可以留在当前状态文件或本地工作日志里。

当观察经过复现或结果验证，我们才得到一条经验：在什么条件下发生了什么，哪种处理方式有效。

如果这条经验未来会反复影响 AI 的判断，它可以进入 `AGENTS.md`、`CLAUDE.md` 或其他规则入口，成为项目协作约定。

如果规则开始对应一组固定操作，就可以整理成工作流或日常操作检查单。

只有当操作足够稳定、输入输出明确、结果可以检查时，才值得把它变成脚本。再往后，如果触发时机也稳定，才考虑交给持续集成（CI）自动执行。

并不是每条经验都要走完整条路径。

刚完成的最小项目记忆模板也是这样设计的：`PROJECT-STATE.md` 接住会变化的当前信息，`AGENTS.md` 只接住稳定的工作规则，`README.md` 则保留项目入口和可信源头。三种内容不挤在同一个文件里，也不要求每条观察都永久留下。

## 4. 有些经验应该晋升

这个系列里已经发生过几次真实的“经验晋升”。

第一次和文章之间的钩子有关。

第二篇结尾预告的是“最小项目记忆系统”，第三篇却先讨论“为什么 AI 写作不显 AI 味”。单独看两篇都没有问题，连续阅读时却会产生断线感。

我们先修正了文章之间的过渡，然后意识到：这不是只会影响一次的文字问题。只要系列继续写，新篇就可能再次和上一篇留下的预告冲突。

于是这条经验进入了 `AGENTS.md`：写系列文章的新篇章时，要先回顾上一篇末尾的钩子、预告或开放问题。它来自真实问题，经过连续阅读视角确认，未来还会重复，所以值得成为项目规则。

第二次和文章状态有关。

早期只在对话里说“这篇可以发布了”，人能理解，脚本却不能可靠读取。后来文章开头增加了结构化状态：`draft`、`review`、`ready`。其中 `status: ready` 不只是一个备注，它同时决定 README 是否收录文章、Wiki 同步脚本是否处理文章。

这条经验没有停在文字规则里，而是变成了人、AI 和脚本都能读取的结构化状态。

第三次和 README 索引有关。

每次文章变化后手动维护中英文 README，很容易遗漏。我们先在 `AGENTS.md` 里写下“修改文章后运行索引脚本”的规则，再用 `scripts/update_readme_index.py` 生成已发布文章区块。

这里要区分两种自动化程度：README 索引目前仍需要按规则运行脚本；文章推送后的 Wiki 发布，则已经交给 GitHub Actions 自动执行。规则、脚本和 CI 不是同一个阶段，也不需要一次全部做完。

这几个例子的共同点，不是它们听起来重要，而是它们都解决了已经发生、可以验证、未来还会重复的问题。

## 5. 有些经验只适合停留在观察层

也有一些观察值得记下来，却不适合晋升为系统规则。

Gitee Wiki 的页面排序就是一个例子。

我们发现 Gitee 左侧导航中的页面可以手动拖拽。既然能调整顺序，很自然会猜测：这个顺序是不是记录在某个 Wiki 文件里，能不能被自动同步？

但实际 push 之后，页面顺序又发生了变化。检查 Wiki Git 仓库时，也没有找到可以稳定控制拖拽顺序的文件。这个排序很可能依赖平台内部状态，而不是主仓库能够版本管理的内容。

这段经历值得保留，因为它能帮助以后排查“为什么手动顺序失效”。但它不适合成为系统依赖：我们无法稳定读取它，也无法通过提交记录检查它是否变化。

最后采用的方案，是给 Wiki 页面文件名加上 `01-`、`02-`、`03-` 这样的序号前缀。

这个方案不一定最漂亮，却满足三个条件：可以版本化，可以在同步前检查，也可以同时作用于 GitHub 和 Gitee。它因此从一次尝试变成了稳定方案。

这个对比让我更清楚地看到：被观察到，不等于可以被依赖。无法验证的现象可以留作经验，但不要轻易写成后续必须遵守的规则。

## 6. 关键不是记住，而是放对位置

判断一条经验是否值得沉淀，只完成了一半。另一半是把它放在正确的位置。

- 当前正在做什么，放进 `PROJECT-STATE.md` 或本地工作日志。
- 长期有效的协作边界，放进 `AGENTS.md`、`CLAUDE.md` 等规则入口。
- 需要反复照着执行的步骤，放进工作流文档或日常操作检查单。
- 输入输出稳定、结果可以验证的机械动作，交给脚本。
- 触发时机明确、失败可以被发现的脚本，再考虑接入 CI。
- 无法稳定验证的平台现象，只保留为观察或排障经验。

同一条信息也不应该在每个文件里复制一遍。

比如文章是否可以发布，可信源头是文章元信息中的 `status`。README 和 Wiki 都应该根据它生成，而不是分别维护另一套状态。否则三个地方迟早会出现三个答案。

所谓“放对位置”，本质上是在回答两个问题：谁需要读取它，以及谁负责更新它。

如果这两个问题说不清楚，即使内容被保存下来，也很可能在未来变成过期信息。

## 7. 给每次沉淀留一道最小检查

以后再遇到一段“感觉应该记下来”的经验时，我会先用五个问题做最小检查：

1. 它解决了哪个真实发生的问题？
2. 有什么证据说明这个判断成立？
3. 它未来还会重复影响工作吗？
4. 放在哪个位置最容易被正确读取？
5. 以后由谁、在什么时机更新或删除？

如果前面三个问题答不上来，它更适合暂时停留在观察层。如果后面两个问题答不上来，说明它还没有找到稳定的承载位置。

这道检查也不是只负责“晋升”。

长期系统还需要降级和清理。项目阶段变化后，旧规则可能不再成立；某个脚本被替代后，对应的操作说明应该删除；一次临时排障结束后，状态文件也不必永久保留所有过程。

真正健康的项目记忆，不是内容只进不出，而是重要经验能进入正确位置，过期信息也能被及时移走。

这样看，从聊天记录到项目规则，中间真正发生的不是一次复制，而是一次判断：什么值得长期影响未来的工作。

当一条规则已经稳定，并且开始被反复执行，下一个问题也就出现了：

> 它什么时候应该继续变成工作流？

这也是从提示词（prompt）走向工作流（workflow）时，真正需要跨过的一步。
