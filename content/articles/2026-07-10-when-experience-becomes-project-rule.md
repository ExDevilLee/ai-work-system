# 从聊天记录到项目规则：一次经验什么时候值得沉淀？

<!--
开篇承接第四篇最后的问题：这件事以后还会重复吗？如果会，它应该沉淀到哪里？
补充模板制作过程带来的新问题：文件已经有了，但不能把每次对话都塞进去。
明确本文主张：沉淀不是保存动作，而是一套筛选和晋升过程。
-->

## 1. 不是每段有用的对话都应该变成长期规则

<!--
区分“当下有用”和“长期有效”。
说明未经筛选地积累会制造冲突、过期信息和维护负担。
承接第四篇“记忆不是越多越好”，但不再展开文件清单。
-->

## 2. 一次经验先经过四个问题

<!--
依次展开四个判断：
1. 是否来自真实发生的问题，而不是预想出来的完美流程？
2. 是否经过验证，而不是临时猜测？
3. 未来是否可能重复影响工作？
4. 应该停留在哪一层：观察、状态、规则、工作流还是脚本？
强调四个问题不是审批流程，而是防止过度沉淀的最小检查。
-->

## 3. 从观察到脚本，不是一步跳过去

<!--
给出晋升路径：临时观察 -> 已验证经验 -> 项目规则 -> 工作流 -> 自动化脚本。
解释并非所有经验都要走到最后：
- 一次性状态留在 PROJECT-STATE 或工作日志。
- 稳定边界进入 AGENTS.md。
- 重复步骤进入操作指南或工作流。
- 稳定、机械、可验证的动作才适合脚本化。
引入刚完成的最小项目记忆模板作为具体承载方式。
-->

## 4. 有些经验应该晋升

<!--
用三个真实案例呈现不同晋升层级：
- 第二篇结尾与第三篇主题冲突：真实发生、经读者视角确认、未来会重复，因此进入 AGENTS.md，成为系列文章钩子规则。
- status: ready：发布状态需要被人、AI 和脚本共同读取，因此从聊天结论变成结构化文章元信息。
- README 索引容易遗漏：先形成修改文章后更新索引的规则，再因为动作稳定且可验证，升级为 update_readme_index.py 和 hook/CI。
重点写“为什么晋升”，不重复技术实现细节。
-->

## 5. 有些经验只适合停留在观察层

<!--
完整使用 Gitee Wiki 拖拽排序案例：页面可拖拽，但 push 后顺序变化，平台状态不可版本化、不可稳定验证。
说明它值得被记录为排障经验，却不适合成为系统依赖。
对照最终采用的文件名前缀：可版本化、可检查、可同步，因此可以进入稳定方案。
-->

## 6. 关键不是记住，而是放对位置

<!--
形成简洁的落点判断：
- 当前进展 -> PROJECT-STATE.md 或本地工作日志。
- 长期协作边界 -> AGENTS.md / CLAUDE.md 等规则入口。
- 重复操作 -> 工作流或日常操作检查单。
- 稳定机械动作 -> 脚本和 CI。
- 无法验证的平台现象 -> 观察或排障记录，不升级为依赖。
说明同一内容不要在多个入口重复复制，保留一个可信源头。
-->

## 7. 给每次沉淀留一道最小检查

<!--
提供读者可直接使用的五问检查单：
1. 它解决了哪个真实问题？
2. 有什么证据说明判断成立？
3. 未来还会重复吗？
4. 放在哪个位置最容易被正确读取？
5. 以后由谁、在什么时机更新或删除？
回扣长期系统也需要清理和降级，不是只会晋升。
结尾钩子：当一条规则已经稳定，并且开始被反复执行，它什么时候应该继续变成工作流？
为第六篇“从提示词到工作流”铺路。
-->
