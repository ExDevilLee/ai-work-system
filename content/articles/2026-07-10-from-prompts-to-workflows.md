# 从提示词到工作流：怎么让 AI 不再每次等你提醒？

上一篇文章最后，我留下了一个问题：

> 当一条规则已经稳定，并且开始被反复执行，它什么时候应该继续变成工作流？

这个问题来自一个很具体的感受。

一开始和 AI 协作时，我会说：“帮我更新这篇文章。”后来任务变复杂，我的提醒也越来越长：“改完文章记得更新 README，检查发布状态，再确认 GitHub Wiki 和 Gitee Wiki 的顺序没有乱。”

如果每次都要重新说一遍，AI 即使执行得很好，协作仍然停留在一次性提示词阶段。

后来我们把一些提醒写成项目规则，又把稳定步骤做成脚本，再把触发时机明确的部分交给 GitHub Actions。AI 不是真的学会了主动想起一切，而是项目开始能够告诉它：什么时候该做什么，完成后怎样证明结果正确，遇到异常又该停在哪里。

这就是我理解的“从提示词到工作流”。

## 1. 当同一句提醒反复出现，问题就不只是提示词

提示词很适合解决一次任务。

“帮我润色这段话”“检查这个脚本”“把文章同步到 Wiki”，只要当前上下文足够清楚，AI 就可以立即开始。

但当同一句提醒不断出现，它通常已经不再属于某一次任务。

比如这个项目每次修改文章后，都可能涉及几件事：

1. 更新中英文 README 的文章入口。
2. 确认只有 `status: ready` 的文章进入发布范围。
3. 检查 Wiki 页面顺序和链接。
4. 推送后同步 GitHub Wiki 和 Gitee Wiki。

如果这些步骤只存在于我的提醒里，每次协作都要重新拼装。少说一句，就可能漏一步；换一个会话，又要从头解释。

反复提醒其实是一个信号：这件事已经具备了固定结构，应该从对话里移出来。

它可能先进入项目规则，也可能进一步变成工作流。判断标准不是提醒有多长，而是它是否会重复、步骤是否稳定、结果能不能检查。

## 2. 规则告诉 AI 应该做什么，工作流还要说明怎么完成

提示词、规则和工作流很容易被混在一起。

提示词解决的是“这一次做什么”。

项目规则解决的是“以后遇到同类情况应该遵守什么”。比如这个仓库的 `AGENTS.md` 规定：文章新增或修改后，要运行 README 索引更新脚本。

但这仍然不是完整工作流。

工作流还要继续回答：

1. 什么变化会触发这件事？
2. 哪份内容是唯一可信输入？
3. 操作应该按什么顺序执行？
4. 怎样判断结果正确？
5. 失败时应该重试、回退，还是交给人确认？

所以，工作流不是把提示词写得更长，也不是把所有步骤塞进一段规则。

它是一条可以重复走通的路径。路径里可以有 AI、脚本、持续集成，也可以保留人工检查点。关键不是“全自动”，而是每一步的职责和边界都清楚。

## 3. 一个最小工作流需要五个部分

如果把复杂工具先放在一边，一个最小工作流至少需要五个部分。

### 触发条件

先说明什么时候开始。

可能是“文章文件发生变化”，也可能是“人确认文章可以发布”，或者“代码推送到 `main` 分支”。没有明确触发条件，工作流仍然要靠人临时想起。

### 可信输入

再说明应该相信哪份内容。

这个项目把 `content/articles/*.md` 作为唯一内容源头，Wiki 只是展示层。这样同步时不会在主仓库、GitHub Wiki 和 Gitee Wiki 之间猜哪份才是最新版。

### 执行步骤

步骤要有顺序，也要能拆开检查。

例如先更新 README 索引，再用试运行（dry-run）检查 Wiki 发布范围，最后才提交并推送。顺序明确后，AI 不需要每次重新设计流程。

### 验证结果

执行完成不等于结果正确。

索引脚本需要确认发布文章数量，Wiki 试运行需要列出准确的文章标题，脚本还要通过语法检查。没有验证的自动化，只是更快地执行一个可能错误的动作。

### 失败处理和人工确认

最后要说明哪里可以自动继续，哪里必须停下来。

网络失败可以重试，脚本语法错误应该立即终止；文章是否达到可发布质量，则要等人审阅。工作流只有知道什么时候停，才算真正有边界。

## 4. 这个项目的发布链路是怎么长出来的

这套发布链路不是一开始就设计完整的。

最早只有文章文件和人工提醒。文章写完以后，我需要记得更新 README，再手动处理展示页面。

真实运行几次后，重复步骤逐渐被拆出来。

首先，文章状态进入了结构化元信息。`draft` 表示草稿，`review` 表示等待审阅，`ready` 表示可以进入发布链路。人和脚本开始共享同一个状态。

然后，项目规则写进了 `AGENTS.md`：修改文章后要更新中英文 README；Wiki 只同步 `ready` 文章；展示层不能反向修改源文。

接着，机械步骤被做成脚本：

- `scripts/update_readme_index.py` 根据文章状态生成 README 索引。
- `scripts/sync_wiki.py --dry-run` 在发布前列出将要同步的文章。
- `scripts/sync_wiki.py --push` 生成 Wiki 页面并推送远端。

最后，触发时机足够稳定的部分进入了 GitHub Actions。

内容或代码推送到 `main` 后，一条流水线同步 GitHub Wiki；另一条流水线把主仓库镜像到 Gitee，并发布 Gitee Wiki。这里的持续集成（CI），可以理解为代码或内容发生指定变化后，由平台自动运行预先约定的检查和脚本。

但这条链路并没有把所有事情都自动化。

README 索引仍然需要在提交前按规则运行；文章也只有在人确认并把状态改为 `ready` 后才会发布。自动化接住的是稳定执行，不是内容判断。

## 5. 脚本和 CI 不是工作流的全部

看到脚本和 GitHub Actions，很容易把工作流理解成“自动跑几个命令”。

但脚本只负责执行一个相对确定的动作，CI 只负责在确定的时机触发它们。

真正的工作流还包括脚本外面的东西：

1. 为什么这次任务应该触发发布？
2. 哪个状态允许文章进入发布范围？
3. 脚本输出什么结果才算通过？
4. 哪类失败可以自动重试？
5. 哪个节点必须等待人的判断？

如果这些问题没有答案，自动化越快，错误扩散得也越快。

比如同步脚本可以忠实地发布所有 `ready` 文章，却无法判断一篇文章是否真的讲清楚了。它也可以把同一份错误内容同时推到两个 Wiki，却不会因为“两个平台都成功”就让内容变得正确。

所以，工作流的价值不只是省掉点击和命令。更重要的是，它把触发、执行、验证和停止条件放进同一个闭环里。

## 6. 有些判断不应该交给自动化

从提示词走向工作流，不等于尽可能减少人的参与。

至少有几类判断，我现在仍然希望留给人：

1. **选题判断**：下一篇写什么，要同时看真实实践、系列钩子和读者价值。
2. **内容审阅**：文章是否表达准确、是否重复、是否照顾第一次阅读的人，不能只靠状态字段判断。
3. **体验检查**：GitHub 和 Gitee 都显示“同步成功”，不代表页面顺序、链接和阅读感受一定舒服。
4. **风险边界**：涉及敏感信息、外部账号、不可逆操作时，自动化不应该替人做最终决定。

这些人工节点不是工作流不完整的表现，反而是工作流的一部分。

自动化最适合接住确定性高、重复性强、结果可以验证的动作。人的价值则更多地出现在目标选择、例外处理、质量判断和风险确认上。

标题里说“让 AI 不再每次等你提醒”，并不是让 AI 从此什么都自己决定。

更准确地说，是让人只在真正需要判断的位置出现，而不是每次都重新提醒那些已经稳定的步骤。

## 7. 一条规则什么时候值得变成工作流

以后再遇到一条反复执行的规则时，我会先问五个问题：

1. 这件事是否已经重复发生，而不是只做一次？
2. 执行步骤是否足够稳定，不需要每次重新设计？
3. 输入和输出是否明确，能找到唯一可信源头？
4. 结果是否可以验证，而不只是“看起来跑完了”？
5. 失败是否能够被发现，并且知道该停在哪里？

如果这些问题大多能回答，就可以开始把规则整理成工作流。

第一步不一定是写代码。先写一份操作检查单，真实跑几次，确认步骤和边界稳定，再决定哪些部分值得脚本化。脚本稳定之后，才考虑是否需要 CI 自动触发。

这样做的好处，是自动化来自已经验证的工作方式，而不是来自对“全自动”的想象。

从提示词到工作流，真正减少的不是人的存在，而是人的重复提醒。

当执行开始交给系统后，人的注意力会转向另一个更重要的问题：

> AI 已经按工作流完成了任务，但我们凭什么确认这个结果真的可以接受？

这也是下一篇值得继续讨论的主题：为什么 AI 工作系统需要审阅（Review），而不只是生成。
